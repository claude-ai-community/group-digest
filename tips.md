## 把 claude.md 里的条件 section 抽成 skill,给它瘦身

- **场景**:claude.md 越写越大,占空间又难维护,里面很多是"特定条件才生效"的内容。
- **做法**:让 Claude Code 自动把这些 conditional section 抽取出来,拆成独立的 skill,实现"渐进式披露"。
- **效果 / 坑**:claude.md 明显瘦身,条件逻辑模块化、好维护;⚠️ 别乱装网上未经验证的第三方 skill,有安全和性能风险。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用终端驱动 Agent 工程,维持深度心流

- **场景**:同时指挥多个 AI 智能体协同工作,容易被界面切换打断心流。
- **做法**:采用"船长-船员"思路——WezTerm(终端)+ tmux(切窗格/跨设备保会话)+ Neovim(编辑),全程手不离键盘。
- **效果 / 坑**:出自前 Meta/微软 L8 工程师 Kun Chen 的分享;群里已有人在看原视频、计划结合自己惯用的 opsx-superpowers 流程。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 多个 AI 工具用 symlink 共享同一份记忆文件

- **场景**:Claude Code 和其他 AI 工具各自维护一份项目记忆(代码规范、架构说明),内容重复、改一处要改多处。
- **做法**:记忆文件分两层(全局个人偏好 + 项目级规范),用 symbolic link 让不同 AI 工具指向同一份文件。
- **效果 / 坑**:一处更新、处处生效。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用 Claude Design 从想法到 UI 成品,别拘泥于"亲手写代码"

- **场景**:想快速验证一个 UI 想法(如财务看板)。
- **做法**:直接用 Claude Design 从想法搭到成品,不追求手写每一行代码。
- **效果 / 坑**:不到一小时出效果;引申心得——应用是为现实服务的,不必拘泥于编程本身。
- 来源:[digests/2026-07-07.md](digests/2026-07-07.md)

## 工匠精神打磨一遍,存成设计系统,后面让 AI 复用

- **场景**:亲自抠细节(工匠精神)和追求 AI 效率看似矛盾。
- **做法**:第一遍自己和 AI 一起抠细节、存成设计系统,后续任务让 AI 照着系统复用。
- **效果 / 坑**:人工判断的"品味"被沉淀下来,同时不用每次从零抠细节。
- 来源:[digests/2026-07-07.md](digests/2026-07-07.md)

## 从"写代码的人"转成"工程主管",别审每个 diff

- **场景**:AI 生成的 diff 太多,亲自审阅每一个,人累成瓶颈。
- **做法**:心态转成 Director——配合自动化流水线(如 No Mistakes),只在系统标记高风险改动时才深度介入审阅。
- **效果 / 坑**:是让"一人管多个 Agent"规模化下去的关键心态转变。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用 NotebookLM 追 guru 的零散长文,省时间抓核心观点

- **场景**:想跟踪行业 guru(如 Martin Fowler)的最新思考,但材料零散、篇幅长。
- **做法**:把原始博客碎片丢给 NotebookLM 做总结,再人工梳理成核心观点。
- **效果 / 坑**:大幅省时间,能快速判断要不要深入读原文。
- 来源:[digests/2026-07-08.md](digests/2026-07-08.md)
