## 把 claude.md 里的条件 section 抽成 skill,给它瘦身

- **场景**:claude.md 越写越大,占空间又难维护,里面很多是"特定条件才生效"的内容。
- **做法**:让 Claude Code 自动把这些 conditional section 抽取出来,拆成独立的 skill,实现"渐进式披露"。
- **效果 / 坑**:claude.md 明显瘦身,条件逻辑模块化、好维护;⚠️ 别乱装网上未经验证的第三方 skill,有安全和性能风险。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用终端驱动 Agent 工程,维持深度心流

- **场景**:同时指挥多个 AI 智能体协同工作,容易被界面切换打断心流。
- **做法**:采用"船长-船员"思路——WezTerm(终端)+ tmux(切窗格/跨设备保会话)+ Neovim(编辑),全程手不离键盘。
- **效果 / 坑**:出自前 Meta/微软 L8 工程师 Kun Chen 的分享;群里已有人在看原视频、计划结合自己惯用的 opsx-superpowers 流程。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 多个 AI 工具用 symlink 共享同一份记忆文件

- **场景**:Claude Code 和其他 AI 工具各自维护一份项目记忆(代码规范、架构说明),内容重复、改一处要改多处。
- **做法**:记忆文件分两层(全局个人偏好 + 项目级规范),用 symbolic link 让不同 AI 工具指向同一份文件。
- **效果 / 坑**:一处更新、处处生效。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用 Claude Design 从想法到 UI 成品,别拘泥于"亲手写代码"

- **场景**:想快速验证一个 UI 想法(如财务看板)。
- **做法**:直接用 Claude Design 从想法搭到成品,不追求手写每一行代码。
- **效果 / 坑**:不到一小时出效果;引申心得——应用是为现实服务的,不必拘泥于编程本身。
- 来源:[digests/2026-07-07.md](digests/2026-07-07.md)

## 工匠精神打磨一遍,存成设计系统,后面让 AI 复用

- **场景**:亲自抠细节(工匠精神)和追求 AI 效率看似矛盾。
- **做法**:第一遍自己和 AI 一起抠细节、存成设计系统,后续任务让 AI 照着系统复用。
- **效果 / 坑**:人工判断的"品味"被沉淀下来,同时不用每次从零抠细节。
- 来源:[digests/2026-07-07.md](digests/2026-07-07.md)

## 从"写代码的人"转成"工程主管",别审每个 diff

- **场景**:AI 生成的 diff 太多,亲自审阅每一个,人累成瓶颈。
- **做法**:心态转成 Director——配合自动化流水线(如 No Mistakes),只在系统标记高风险改动时才深度介入审阅。
- **效果 / 坑**:是让"一人管多个 Agent"规模化下去的关键心态转变。
- 来源:[digests/2026-07-06.md](digests/2026-07-06.md)

## 用 NotebookLM 追 guru 的零散长文,省时间抓核心观点

- **场景**:想跟踪行业 guru(如 Martin Fowler)的最新思考,但材料零散、篇幅长。
- **做法**:把原始博客碎片丢给 NotebookLM 做总结,再人工梳理成核心观点。
- **效果 / 坑**:大幅省时间,能快速判断要不要深入读原文。
- 来源:[digests/2026-07-08.md](digests/2026-07-08.md)

## 用 NotebookLM 做深度追问,而不只是总结

- **场景**:看完一个视频/演讲后,冒出一个视频没直接回答的深层问题(比如"AI 时代的水轮是什么"),自己想不透。
- **做法**:把这个追问连同原视频一起丢给 NotebookLM,让它结合视频内容做结构化推演,而不是简单摘要。
- **效果 / 坑**:能得到比自己空想更系统的类比框架,NotebookLM 会顺着视频的逻辑往下推,而不是另起炉灶。
- 来源:[digests/2026-07-14.md](digests/2026-07-14.md)

## AI 生成代码合规,五层防御挡住"撞公开代码"风险

- **场景**:担心 AI 生成的代码撞到公开代码,有版权/许可证合规风险,尤其在受监管行业。
- **做法**:① 模型层选带查重过滤的工具(如 GitHub Copilot 的 Duplication Detection)② Prompt 层改成描述业务约束而非"实现 X 算法" ③ 后处理层用 Semgrep/Black Duck 扫描 + GitHub code scanning 查许可证风险 ④ 架构层区分高低风险(算法题/经典数据结构 vs 业务逻辑/领域代码)⑤ 合规层看企业协议条款。
- **效果 / 坑**:受监管行业实测有效——公司不让改模型设置的情况下,靠后处理扫描 + wrapper 就能过关。
- 来源:[digests/2026-07-09.md](digests/2026-07-09.md)

## VibeCoding 落地合规,五步扫描组合拳

- **场景**:AI 高吞吐生成代码上量后,担心代码库像 Bun 案例那样(6778 次提交没人完整读过)埋下维护成本和安全隐患。
- **做法**:人工审查 + bugbot 等 AI 审查 + SBOM 供应链扫描 + GitHub 代码/镜像安全扫描 + DAST 和 PenTest 扫描,五层叠加。
- **效果 / 坑**:SOC2 等合规要求下这些已是必备项,不是可选项。
- 来源:[digests/2026-07-09.md](digests/2026-07-09.md)

## Harness 的核心就两点:定标准、检查是否符合标准

- **场景**:想让 Agent 像 Claude Code 一样可控地完成复杂任务,而不是把大模型一次性回答直接扔给用户。
- **做法**:做之前先定义标准,做了之后检查是否符合标准;Agent 落地成多步骤、多 agent 的工作流,检查环节可以包含合规检查,找不到答案就不输出,而不是让 AI 自己瞎猜。
- **效果 / 坑**:知识库怎么用、答案能不能验证、给答案前能不能根据知识库做检验,这些都得靠机制,不能靠碰运气。
- 来源:[digests/2026-07-10.md](digests/2026-07-10.md)

## Plan 阶段别写死 if-then,把判断交给大模型

- **场景**:Agent 的 plan 和执行之间,该写死 if-then 规则,还是让大模型自己判断?
- **做法**:不定死 if 条件,把判断这件事交给模型;人该做的是定义结果、检查结果,把执行交给 AI。
- **效果 / 坑**:以前的决策树在条件模糊时容易判断错;现在的大模型判断力往往比人自己写的 if 条件更准——自己写 if-then 本质是自己在做执行,而这部分恰恰是 AI 更擅长的。
- 来源:[digests/2026-07-10.md](digests/2026-07-10.md)

## 处理大量 word/PDF,先转 markdown 再放 project 里更省 token

- **场景**:想省 token 又要处理大量 word/PDF 文档。
- **做法**:先用 Markitdown skill 把文档转成 markdown,一次转换后放进 project 里复用,而不是每次都用 Claude 内置工具读原始文件。
- **效果 / 坑**:更省 token;但也有人提醒——先别急着优化 token,先图省事省力,等真的 scale up 了再考虑效率。
- 来源:[digests/2026-07-11.md](digests/2026-07-11.md)

## 写 Prompt 只给验收标准和边界,不给 how

- **场景**:怎么写出好的 Prompt,而不是手把手教模型解题步骤。
- **做法**:① 不写解题步骤,只写清楚"什么叫解完" ② 把定义、范围和边界情况一次性说清楚 ③ 不只告诉模型什么算答案,还要列出什么不算答案 ④ 复杂任务不固定分工,而是动态搜索,并设置独立审查。
- **效果 / 坑**:本质是"只给验收标准和边界,不给 how"(即 /goal 思路);未来 agent 可能会变薄,graph flow 这类编排会被大模型自己端到端接管。
- 来源:[digests/2026-07-11.md](digests/2026-07-11.md)

## 把 AI 当"有才华但没经验的员工"管理,而不是全权信任

- **场景**:AI 能力"锯齿状"(Jagged Frontier)——某些任务远超人类,但跨任务协调、处理突发异常时很脆弱,且它没有"切肤之痛",做错了不承担后果。
- **做法**:决策权和后果承担权留在人手里,AI 只负责执行;通过 Verification loop(验证循环)持续检查输出,而不是一次性全权信任它的判断。
- **效果 / 坑**:呼应 [7/14 Austin 的锯齿状前沿框架](digests/2026-07-14.md)——谁承担后果,谁就该保留最终决策权。
- 来源:[digests/2026-07-16.md](digests/2026-07-16.md)

## Spec 写成"三分钟审得动"的极限压缩版

- **场景**:AI 编程前想先写清楚 spec,但 spec 写太啰嗦,没人愿意认真审。
- **做法**:spec 压缩成"架构设计的极限压缩版"——一屏摘要 + 关键图 + 契约/不变量 + 验收标准,别的细节让 AI 自己在实现里填。
- **效果 / 坑**:三分钟就能审完,呼应 Harness"先定标准、再检查是否符合标准"的思路,让 spec 本身变得可执行、可审查。
- 来源:[digests/2026-07-18.md](digests/2026-07-18.md)

## 比模型贵不贵,要看"每完成一个任务的成本",不能只看 token 单价

- **场景**:纠结用哪个模型更省钱(比如 Sonnet 5 单价比 Opus 4.8 便宜)。
- **做法**:按"每完成一个任务的成本"衡量,而不是单纯比 token 单价——同时看完成率和实际消耗的 token 数。
- **效果 / 坑**:Databricks 研究发现 Sonnet 5 单价更低,但完成率较低(81% vs 87%)且消耗更多 token,算下来实际每任务成本反而更高。
- 来源:[digests/2026-07-18.md](digests/2026-07-18.md)

## 本地跑模型编程,先看硬件门槛,再看模型质量

- **场景**:想本地部署模型做编程任务,不确定自己的硬件能不能扛。
- **做法**:按模型规模选硬件——装最新 GLM 5.2 可能要好几张 RTX 5090/6000 GPU 或 H100(几万美元);装 Qwen 3.6 的 27B/35B 模型,消费级 Mac Studio 或 NV 的 DGX 就够,35B-A3B 能到 70 token/s,大多数编程任务能跑。
- **效果 / 坑**:27B/35B 能用,但复杂任务仍需云模型;实测"本地模型 Vibe Coding"体验不佳——免费的 OSSGPT 120B(号称与 Qwen 3.5 相当)"惨不忍睹",对 Qwen 3.6/3.7 能不能有质的提升持怀疑态度。
- 来源:[digests/2026-07-18.md](digests/2026-07-18.md)
