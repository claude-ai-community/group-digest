## 心得

- **场景**:担心 AI 生成的代码撞到公开代码,有版权/许可证合规风险。
  **做法**:五层防御——① 模型层选带查重过滤的工具(如 GitHub Copilot 的 Duplication Detection)② Prompt 层改成描述业务约束而非"实现 X 算法",越具体越偏离公开代码 ③ 后处理层用 Semgrep/Black Duck 扫描生成结果,配合 GitHub code scanning 查许可证风险 ④ 架构层区分风险:算法题/经典数据结构高风险,业务逻辑/领域特定代码低风险 ⑤ 合规层看 OpenAI/Anthropic 的企业协议条款。
  **效果 / 坑**:T 在受监管行业实测——公司不让改模型设置,靠后处理扫描就够了,AI 自己定位到问题所在,加个 wrapper 就过了。
  贡献:Austin(方案)+ T(验证反馈)

- **场景**:VibeCoding 高吞吐生成代码后,团队担心维护成本失控(引 Bun 案例:Claude 11 天重写完 Bun,创始人一个月才敢交底,代码库 6778 次提交没一个人完整读过)。
  **做法**:Y 分享自己团队的五步合规实践——人工审查 + bugbot 等 AI 审查 + SBOM 供应链扫描 + GitHub 代码/镜像安全扫描 + DAST 和 PenTest 扫描。
  **效果 / 坑**:VibeCoding 已不可避免、甚至可能完全取代传统人工编码,SOC2 等合规要求这些已是必备项;团队正在和厂商谈引入新技术。
  贡献:Y

## 讨论

- **AI 与组织变革,续集**:D 分享《裁员 12 万人后,55% 雇主后悔:AI 无法替代人类的"组织知识负债"》([链接](https://mp.weixin.qq.com/s/608-vbwYvkxI6mrCtzK0bg)),讨论由此展开。Austin 观点:AI 是生产力变革,必然带来生产关系变革,不是简单裁员,而是改造为面向 AI 的组织——新组织可能不按人头数衡量,现有大企业若只是裁员 + 上 AI 可能人财两失。Y 观点:裁员是对的,否则怎么降薪?现在是劳动力买方市场,不愁招不到人,老人思维难转变。Austin 回应:裁员再招有风险,裁到大动脉的情况也时有发生,用 AI 来内卷人不可取。GK 用汽车取代马车类比:基础设施和人的观念转变都需要时间。

- **AI 生成代码到底要不要人审?**(本期最长、最激烈的一场辩论,从 T 的合规提问一路烧到凌晨 3 点):起因是 T 问如何让 AI 避免生成 public matched code(见心得五层防御方案)。方案验证有效后,Austin 抛出更根本的问题:智能体不是魔法程序员,是生产线上的高吞吐执行单元([链接](https://mp.weixin.qq.com/s/T-Er4yhYR0Vx5Hui8nNUIA));引用 Bun 案例([链接](https://mp.weixin.qq.com/s/H26YZeHvAU3kaDKYA6PDcQ))追问:六个月后并发问题冒出来怎么办、维护成本怎么算?如果都是 AI 写代码,还要人来审核吗?一种说法是人只要管 spec(自然语言需求),另一种说法是人还是得看代码但成了瓶颈——但像 T 遇到的合规/许可证问题,人不看谁把关?若人还要看,AI 的高生产率就被约束,人依然在 loop 中。群友立场分化明显:Q 认为互联网行业习惯了 screw up 然后回滚,没惩罚就没动力提质量(举小米 SU7 为例);G 认为不可能靠人来 review code,写了二三十年代码也未必写得过 AI;T(监管行业视角)认为对监管行业尤其重要,AI 生成的 spec 和 code 都得在信任度范围内,关键数据分析的 validation 是必须的;KY 反映监管项目即便批准用 AI,还是得用传统笨办法写流程以保证每一步可控,加了 restrictions/guardrails 仍不放心,宁可体验差也不能碰 regulations,AI 能快速出 demo 但一到真实项目就歇菜,工作反而更忙了;L 分享自己的折中做法——用 Claude 写、用 ChatGPT 审,自己已二三十年没碰代码。Austin 总结:说到底是 review 的人或工具是否足够强大——人之所以要 review,是因为现在人比 AI 更 knowledgable、能发现 AI review 发现不了的问题;真正需要的是足够强大的 evaluation tool,人没法短时间内跑遍所有 user case。Y 借此分享了自己团队的五步合规实践(见心得),Austin 回应让 T 去联系 Y,两人还是校友。
